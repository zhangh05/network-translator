# Repository Hygiene

## Category Definitions

| Category | Description | Examples |
|----------|-------------|----------|
| **Source** | Actively maintained code | `core/`, `tools/`, `frontend/`, `web_app.py` |
| **Knowledge** | Feature reference data | `knowledge_data/*.md`, `registry.yaml`, `capability_map.yaml` |
| **Test** | Regression and capability verification | `tests/`, `bench/cases/` |
| **Docs** | User-facing and operator documentation | `docs/`, `README.md`, `CHANGELOG.md` |
| **Config** | Build/run/release configuration | `.env.example`, `requirements.txt`, `scripts/`, `VERSION` |
| **Runtime** | Generated during execution (gitignored) | `logs/`, `cache_data/`, `memory_data/`, `projects/`, `venv/` |
| **Legacy** | Superseded but preserved for compatibility | `bench/harness.py`, `knowledge_data/cisco/` (Phase 1 fallback) |
| **Planning** | Historical design documents | `plans/`, `docs/superpowers/` |

## Source vs Runtime Boundary

All files under git tracking are considered **source/knowledge/test/docs/config**.
The following are strictly **runtime-only** and must never be committed:

```
logs/           — service logs + translation JSONL
cache_data/     — LLM response cache
memory_data/    — translation history
projects/       — project persistence
venv/           — virtual environment
.run/           — PID files
__pycache__/    — Python bytecode
.pytest_cache/  — pytest cache
*.pyc           — compiled Python
```

## Hygiene Checklist

Run periodically and before releases:

### 1. Clean Runtime Caches

```bash
./scripts/clean_runtime_artifacts.sh
```

Removes: `__pycache__/`, `*.pyc`, `.pytest_cache/`, `.semantic_cache/`

### 2. Verify .gitignore Coverage

```bash
# Check for any untracked files that look like secrets or runtime data
git ls-files --others --exclude-standard
# Should return empty in a clean checkout
```

### 3. Check for API Key Leaks

```bash
# Verify no API keys in tracked files
git ls-files | xargs grep -l "sk-" 2>/dev/null || echo "No API keys in source"
```

### 4. Verify Working Tree Is Clean

```bash
git status --short
# Should return empty before releases
```

### 5. Run Quality Gate

```bash
PYTHONPATH=. python3 scripts/release_gate.py
# Must return RELEASE_GATE_OK
```

## Managing Legacy Paths

### Knowledge Fallback Paths

`knowledge_data/cisco/`, `huawei/`, `h3c/` are Phase 1 legacy paths.
They are kept for backward compatibility but the `knowledge_data/domains/` path
takes priority. New knowledge files should always go under:

```
knowledge_data/domains/{domain}/{vendor}/{feature}.md
```

### Legacy Benchmark

`bench/harness.py` is deprecated in favor of `bench/run_cases.py`.
Do not add new features to `harness.py`; use `run_cases.py` for new work.

### Planning Documents

`plans/` and `docs/superpowers/plans/` contain historical architecture plans.
They are preserved for traceability but are not active source.

## Security: API Keys

The `llmsetting.json` file may contain sensitive API keys.
It is listed in `.gitignore` and must never be committed.

Use `llmsetting.example.json` as a template — it contains empty placeholder values.

To verify no keys have leaked:

```bash
git ls-files | xargs grep -l "sk-" 2>/dev/null && echo "WARNING: API keys found!" || echo "OK: no keys in tracked files"
```

## .gitignore Maintenance

When adding a new runtime or generated directory, add it to `.gitignore`:

```gitignore
# New runtime directory
path/to/new_runtime_dir/
```

Common additions:
- `*.log` — if creating new log files outside `logs/`
- `*.pid` — if creating PID files outside `.run/`
- `reports/` — if adding report generation

## Keeping the Tree Clean

1. **Do not** commit files generated by running the application
2. **Do not** commit local configuration with real credentials
3. **Do not** commit planning documents to active source directories
4. **Do** keep `plans/` and `docs/superpowers/` for traceability
5. **Do** update `.gitignore` when adding new runtime directories
6. **Do** run `clean_runtime_artifacts.sh` before checking coverage metrics
